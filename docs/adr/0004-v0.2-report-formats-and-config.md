# ADR 0004: v0.2 报告格式、配置与持久化方案

## 状态

已采纳

## 背景

v0.1 MVP 只输出 Markdown 报告，无配置文件，Web UI 扫描结果不持久化。v0.2 需要补齐这三块能力，使工具从"能跑"变为"可交付使用"。

## 决策

### 报告格式

- **JSON**：直接序列化 `ScanResult` 类型，不做中间 schema 转换。类型本身已是稳定的公开契约
- **HTML**：生成自包含静态 HTML 文件（内联 CSS/JS），双击浏览器打开即可。不依赖 Next.js 运行时——报告的价值在于离线分享，对方不需要跑服务
- **SARIF**：输出符合 SARIF 2.1.0 规范的最小有效 JSON。只填充 `runs[0].tool`、`runs[0].results`、`runs[0].invocations`，不做 taxonomies/rules 映射（留到后续版本）
- v0.2 只生成**开发者版**报告（完整 Findings）。客户版通过字段过滤实现，不在 v0.2 范围
- CLI 新增 `--format markdown|json|html|sarif` 参数，默认 `markdown`

### 配置文件

- 文件名：`miniapp-audit.config.json`，放在被扫描项目根目录
- 配置项仅限"扫什么"：`rules`（规则开关）和 `exclude`（排除路径）
- "怎么输出"不写进配置，由 CLI `--format` / `--output` 控制
- 无配置文件时全部规则默认启用

### Web 扫描历史

- 使用浏览器 `localStorage` 持久化。历史列表保存扫描摘要（项目路径、时间、风险统计）
- 为支持跨会话回看报告，同时按 `report id` 保存完整 `ScanResult`（key: `miniapp-audit:report:<id>`）
- 存储上限 50 条，淘汰历史记录时同步删除对应的完整报告
- 无需服务端代码，无数据库依赖，数据仍只保存在浏览器本地，不离开浏览器

## 替代方案

| 决策点 | 采纳方案 | 替代方案 | 不采纳原因 |
|--------|----------|----------|------------|
| JSON schema | 序列化 ScanResult | 定义独立 JSON Schema | 多一层维护成本，无实际受益 |
| HTML 渲染 | 静态文件 | Next.js 动态路由 | 离线交付场景要求自包含文件 |
| SARIF 深度 | 最小子集 | 完整 SARIF（含 taxonomies/rules） | v0.2 无规则元数据体系，盲目映射反而误导 |
| 配置文件内容 | 只控扫描 | 也控输出格式 | 分离关注点：配置管"扫什么"，CLI 参数管"怎么输出" |
| 历史存储 | localStorage | SQLite / JSON 文件 API | v0.2 数据量小，无需引入后端依赖 |
| 客户版报告 | 推迟到后续版本 | 在 v0.2 同时做 | 字段过滤方案未充分验证，先做完开发者版再议 |

## 后果

- `src/scanner/report/` 下新增 `json.ts`、`html.ts`、`sarif.ts` 三个生成器
- `src/scanner/config.ts` 新增配置加载器
- CLI 新增 `--format` 参数和 `--config` 参数
- Web UI 新增扫描历史页面，使用 `localStorage` 读写
- 项目根目录新增 `miniapp-audit.config.json` 作为示例配置
